# 4. 十二条指导原则

**原则 1 先理解问题，再生成解**

| **主张：**没有明确领域边界和业务不变量时，代码生成只能作为探索，不能进入正式交付。 |
|------------------------------------------------------------------------------------|

| AI的角色 | 抽取概念、发现歧义、提出模型候选和问题清单。        |
|----------|-----------------------------------------------------|
| 工程机制 | 领域建模阶段、统一语言、上下文地图、需求—规则追踪。 |
| 需要防止 | 把 DTO、Controller、数据库表直接等同于领域模型。    |

**原则 2 决策必须显式，不允许架构在 Diff 中偷偷发生**

| **主张：**任何改变组件边界、依赖方向、数据所有权或运行拓扑的选择都应形成 ADR。 |
|--------------------------------------------------------------------------------|

| AI的角色 | 生成备选方案，比较成本、风险、可逆性和兼容性。  |
|----------|-------------------------------------------------|
| 工程机制 | ADR 门禁、架构约束清单、决策状态与废弃机制。    |
| 需要防止 | AI 以“顺手重构”为由扩大范围，形成隐式架构漂移。 |

**原则 3 先定义行为，再讨论实现**

| **主张：**验收标准必须能由外部行为描述，而不是由类名、方法名或内部结构描述。 |
|------------------------------------------------------------------------------|

| AI的角色 | 把自然语言需求转成 BDD 场景并发现缺口。     |
|----------|---------------------------------------------|
| 工程机制 | Given-When-Then、场景覆盖矩阵、业务示例库。 |
| 需要防止 | BDD 变成测试代码的逐句翻译，失去业务语言。  |

**原则 4 边界靠契约治理，不靠团队默契**

| **主张：**服务、模块、事件和数据产品之间的承诺必须机器可校验。 |
|----------------------------------------------------------------|

| AI的角色 | 从行为和 ADR 生成契约候选，分析兼容性影响。      |
|----------|--------------------------------------------------|
| 工程机制 | OpenAPI、JSON Schema、Pact、事件版本策略。       |
| 需要防止 | 只做接口冒烟测试，却没有消费者视角和兼容性规则。 |

**原则 5 测试是分层证据，不是数量指标**

| **主张：**测试层级应与风险来源匹配，追求可解释覆盖，而非单纯覆盖率。 |
|----------------------------------------------------------------------|

| AI的角色 | 生成测试矩阵、边界案例和失败解释。                       |
|----------|----------------------------------------------------------|
| 工程机制 | 领域单测、组件测试、契约测试、接口测试、少量端到端测试。 |
| 需要防止 | 大量脆弱 Mock 或重复测试制造虚假安全感。                 |

**原则 6 生成者不能独占裁判权**

| **主张：**写代码的 Agent 不应是唯一的测试设计者和完成判定者。 |
|---------------------------------------------------------------|

| AI的角色 | 不同角色分别生成、审查和归因。               |
|----------|----------------------------------------------|
| 工程机制 | Subagent 隔离、独立 Reviewer、确定性执行器。 |
| 需要防止 | 同一上下文中的模型修改测试以迎合实现。       |

**原则 7 循环必须有边界、预算和终态**

| **主张：**自主循环不是无限重试，而是受控地逼近明确目标。 |
|----------------------------------------------------------|

| AI的角色 | 根据失败证据选择下一动作并提出停止建议。          |
|----------|---------------------------------------------------|
| 工程机制 | 最大轮次、时间/Token 预算、无进展检测、终态枚举。 |
| 需要防止 | 反复重跑相同命令、随机改动、成本失控。            |

**原则 8 从最早错误来源修复，而不是只修最后症状**

| **主张：**接口测试失败可能源于需求、领域、契约、实现或环境，必须先归因再路由。 |
|--------------------------------------------------------------------------------|

| AI的角色 | 对证据进行因果分析并标记置信度。                  |
|----------|---------------------------------------------------|
| 工程机制 | 失败分类、责任路由、回退点、最小修复策略。        |
| 需要防止 | 所有错误都交给 Code Agent，导致契约和测试被污染。 |

**原则 9 Harness 是产品，模型是可替换部件**

| **主张：**可靠性主要来自上下文、工具、权限、反馈、状态与验证的组合。 |
|----------------------------------------------------------------------|

| AI的角色 | 在受控 Harness 中执行认知任务。                            |
|----------|------------------------------------------------------------|
| 工程机制 | 外部 Orchestrator、沙箱、Hooks、权限、状态存储、可观测性。 |
| 需要防止 | 把流程逻辑写死在某个模型的长 Prompt 或聊天历史中。         |

**原则 10 稳定知识进入 Skill，事实状态进入 Artifact**

| **主张：**SOP、规范和失败经验应与单次会话上下文分离。 |
|-------------------------------------------------------|

| AI的角色 | 从成功与失败 Episode 中提炼可复用知识。       |
|----------|-----------------------------------------------|
| 工程机制 | Skill、项目规范、规则库、产物索引、版本管理。 |
| 需要防止 | 未经验证的单次经验自动升级为永久规则。        |

**原则 11 权限随阶段最小化**

| **主张：**领域分析、代码编辑、部署和审查需要不同权限，不应使用一个全能 Agent。 |
|--------------------------------------------------------------------------------|

| AI的角色 | 在授权范围内调用工具并说明需要升级的权限。             |
|----------|--------------------------------------------------------|
| 工程机制 | 只读 Agent、目录级写权限、命令白名单、环境隔离、审批。 |
| 需要防止 | 为减少交互而长期使用危险的全权限模式。                 |

**原则 12 人类负责价值、不可逆决策和例外治理**

| **主张：**AI 可以扩大分析带宽，但不能替代业务价值选择、风险接受和生产责任。 |
|-----------------------------------------------------------------------------|

| AI的角色 | 提供选项、证据、影响分析和建议。                     |
|----------|------------------------------------------------------|
| 工程机制 | 人工审批点、风险分级、责任记录、紧急停止。           |
| 需要防止 | 把不确定性隐藏成自动决策，形成“无人负责”的自主系统。 |

# 5. DDD、ADR、BDD、契约与测试的统一关系

这些方法不是按文档类型机械串行，而是形成相互制约的追踪网络。上游产物给下游提供语义和约束，下游执行结果也可能证明上游理解错误。

| **上游产物**   | **约束下游**                           | **下游反馈可能触发**                                   |
|----------------|----------------------------------------|--------------------------------------------------------|
| DDD 不变量     | 领域单元测试、服务职责、数据所有权     | 测试显示规则矛盾时，回到领域模型或需求澄清             |
| ADR 决策与后果 | 代码结构、依赖方向、技术选型、迁移方式 | 编译/性能/运维证据显示方案不可行时，新增或替代 ADR     |
| BDD 场景       | 契约示例、测试矩阵、接口验收           | 接口行为无法满足或场景不可执行时，修订验收定义         |
| 接口/事件契约  | Provider 实现、Consumer Stub、兼容策略 | Provider/Consumer 验证失败时，判断实现错误还是契约变更 |
| 实现 Diff      | 单元、组件、接口和回归测试             | 失败归因后回到实现、契约、行为或决策层                 |

| **追踪要求：**每条业务规则和验收标准至少应映射到一个验证证据；每项重大架构约束至少应映射到一个静态检查、评审项或测试。 |
|------------------------------------------------------------------------------------------------------------------------|

# 6. Loop、Goal、Skill、Subagent 与 Harness 的分工

| **概念**              | **在本系统中的定义**                                                 | **不应承担的职责**                                     |
|-----------------------|----------------------------------------------------------------------|--------------------------------------------------------|
| Goal / Goal Contract  | 一个可验证目标，包含范围、验收、禁止项、验证方式和终态               | 不能只是“完成某功能”这类无证据目标                     |
| Loop                  | 围绕目标重复执行“选择动作—执行—观察—归因—修复”的外部循环规范         | 不能无限执行，也不能把内部模型思考循环当成完整工程循环 |
| Skill                 | 可版本化、可复用的任务 SOP、项目知识和检查清单                       | 不能替代实时事实、任务状态和最终验证                   |
| Subagent              | 阶段内部的角色隔离和上下文隔离机制                                   | 不负责跨阶段持久状态、全局事务和部署治理               |
| Harness               | 模型之外的运行系统：工具、状态、权限、上下文、反馈、验证、恢复和观测 | 不能把所有判断重新委托给另一个模型                     |
| External Orchestrator | 跨阶段状态机、产物路由、预算、审批和恢复控制器                       | 不承担代码理解等高认知任务                             |

## 推荐边界

- 外部 Orchestrator 维护长期状态、执行顺序、产物、预算、审批和终态。

- Claude Code headless 或 Agent SDK 作为阶段执行引擎，负责读代码、改代码、运行受限命令和生成结构化结果。

- Subagent 用于单阶段内部的探索、实现、独立审查和高噪声日志分析。

- Skill 用于装载项目级 SOP，而不是保存某次需求的动态状态。

- CI、编译器、测试框架、契约工具和 API Runner 提供确定性反馈。
